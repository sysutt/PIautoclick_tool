# 深空图像自动后期处理系统 — 技术方案 v1

> 状态:**草案,待讨论**。本文用于逐节确认;标记 🔶 的是需要你拍板的开放问题。
> 定位:在现有 `AutoClick`(对话框自动点击器)基础上,扩展为"脚本执行 + AI 评估 + 闭环决策"的半自动 DSO 后期处理系统。

---

## 1. 目标与范围

### 1.1 目标
- 把深空图像(DSO)线性数据 → 成片的后期流程,做成**可复现、可无人值守、带质量反馈闭环**的自动化系统。
- 执行层完全脚本化(PixInsight PJSR);评估层用"定量指标 + 多模态大模型"双轨;决策层做闭环编排。
- 屏幕自动点击(现有 AutoClick)退居为"意外模态弹窗守卫",不承担主流程。

### 1.2 非目标(至少 v1 不做)
- 核心范围不含叠加/预处理(校准、配准、Integration)——默认输入已是叠加好的线性主图(masters);
  WBPP 一键叠加作为**可选前置**保留接口(§6.0),但不是 v1 核心。
- 不追求"一键出片吊打人工";目标是**稳定产出 80 分、可复现、省人力**,把人从重复点击里解放出来。
- 不做全自动无监督训练;LLM 只做推理期的评委,不在本阶段做微调。

### 1.3 成功标准(建议,待确认 🔶)
- 对同一批数据,系统能全自动跑完管线并输出成片 + 完整决策日志。
- 相同输入 + 相同配置 → 相同输出(可复现)。
- 在一组测试图上,盲评(人工)结果 ≥ "认真手动处理"的 70–80%。
- 单张总处理时间可接受(目标 🔶,例如 < 30 min/张)。

### 1.4 设计原则(源自作者后期理念)
1. **由问题出发,而非由步骤出发**:每一步都绑定"它解决什么问题"和"怎样算解决好了"(达标标准),而不是机械执行固定顺序。系统的每个环节因此都是"操作 + 评估"的组合。
2. **自动优先,手动兜底**:能自动达标就用自动实现;不达标才升级到更强/更手动的实现(降级阶梯,§6.3)。
3. **真实性优先于美观**:只增强已有信号,绝不凭空添加不存在的结构(§6.5)。
4. **可复现、可回溯**:每步的参数、指标、决策全程落盘。
5. **适配性**:对第三方模块(三件套等)有/无两种情况都能跑,只是能力上限不同(§6.3)。

## 2. 总体架构

```
┌──────────────────────────────────────────────────────────────┐
│  Orchestrator 编排层 (Python)                                   │
│  · 管线状态机   · 数值优化   · LLM介入调度   · 全程日志/复现       │
└───────┬───────────────────────────────────┬───────────────────┘
        │ 下发 job(进程+参数)                 │ 回收 指标JSON + 预览PNG
        ▼                                     ▼
┌───────────────────────────┐        ┌───────────────────────────┐
│ 执行层  PixInsight (PJSR)   │        │ 评估层                      │
│ · 常驻派发脚本 job-runner    │        │ 4.1 定量指标 (确定性算法)    │
│ · 参数化 Process 模板        │        │ 4.2 感知评委 (多模态 LLM)    │
│ · 导出预览(标准化STF)+指标   │        │      图+数 → 离散方向决策     │
└───────────────────────────┘        └───────────────────────────┘
        ▲
        │  旁路,不进主流程
┌───────────────────────────┐
│ AutoClick 弹窗守卫(现有)    │  只处理意外模态框
└───────────────────────────┘
```

### 组件职责一句话
- **执行层**:只负责"按给定参数跑一个 Process、把结果和指标吐出来",无决策。
- **评估层**:把图像变成"可判断的信息"——定量指标(客观)+ 感知判断(主观)。
- **编排层**:唯一有决策权的地方——决定下一步做什么、参数怎么调、何时停。
- **弹窗守卫**:安全网。

## 3. 执行层:PixInsight 脚本化

### 3.1 为什么脚本优先(已达成共识)
PJSR 可精确设参、直接读像素/统计、可命令行运行、可复现;屏幕操作在参数控制、取数、稳定性、复现性上全面劣势。屏幕操作只保留兜底。

### 3.2 运行形态:常驻派发脚本(job-runner)🔶
PixInsight 在 Windows 上并非真正 headless(它是 GUI 程序,通常需要一个交互式会话)。逐步"启动→执行→退出"进程代价高。建议方案:

- 在 PixInsight 内运行一个**常驻 PJSR 脚本**(`job-runner.js`),循环轮询一个 `jobs/` 目录。
- Python 编排器把一个 job 写成 JSON(`{op, params, input, outputs}`)丢进 `jobs/inbox/`。
- job-runner 读取 → 构造对应 Process → `executeOn(view)` → 导出结果图 + 指标 JSON + 预览 PNG → 写 `jobs/done/`。
- Python 监听 `done/` 拿结果。

> 优点:一个长驻 PixInsight 会话服用整条管线,省去反复启动;IPC 简单(文件交换),跨进程稳。
> 🔶 待确认:是否接受"需要一个常驻 PixInsight 会话"这一前提?备选是每步 `PixInsight -r=step.js` 冷启动(简单但慢)。

### 3.3 Process 参数化模板
每个处理步骤封装成一个模板函数,输入结构化参数,输出标准结果。例:
```
// 伪代码
function runHT(view, {shadows, midtones, highlights}) {
  var P = new HistogramTransformation;
  P.H = [[shadows, midtones, highlights, 0, 1], ...];
  P.executeOn(view);
}
```
覆盖的 Process(v1 目标集,见第 6 节):DBE/ABE、BackgroundNeutralization、SpectrophotometricColorCalibration(或 ColorCalibration)、(可选)BlurXTerminator、StarXTerminator、HistogramTransformation/GHS、MultiscaleLinearTransform 或 NoiseXTerminator、CurvesTransformation、SCNR 等。第三方模块(RC-Astro 系列)若安装则同样脚本化。

### 3.4 预览渲染标准化(关键!)
LLM 与人看到的"图"必须可复现、可比较:
- 线性阶段:用**固定参数的 STF AutoStretch**(经典 24-bit:阴影裁剪 −2.8σ、目标背景 0.25)生成预览,**只用于观看/评估,不改动实际数据**。
- 非线性阶段:直接导出当前状态。
- 预览统一导出为固定尺寸(如长边 1024–1536px)PNG,sRGB,附带渲染参数记录。
- 🔶 待确认:预览分辨率、是否给 LLM 看整图 + 若干 100% 裁切块(星点/背景/核心)。

### 3.5 与编排层的数据契约
见第 8 节 JSON schema。原则:执行层无状态、幂等,所有决策信息经文件传递,便于日志与复现。

## 4. 评估层

### 4.1 定量指标模块(确定性,主力)
由 PixInsight 统计或独立算法产出,**不经过 LLM**。用作硬约束与数值优化目标:

| 指标 | 来源 | 主要用途 |
|---|---|---|
| 背景中位数 / 均值 | ImageStatistics | 拉伸目标、背景亮度控制 |
| 噪声 σ(背景区域 stdDev / MAD) | 统计 / MRS 噪声估计 | 降噪强度、SNR |
| 黑场剪切 % / 白场剪切 % | 直方图 | 拉伸是否压死暗部/烧星核 |
| FWHM | SubframeSelector / FWHMEccentricity | 解卷积/锐化效果、是否过度 |
| 星点偏心率 eccentricity | 同上 | 星点畸变监控 |
| 星点数量 | StarDetector | 星点抑制强度、过处理监控 |
| 饱和像素比例 | 直方图 | 高光烧毁监控 |
| 通道背景差(R/G/B 中位数) | 统计 | 偏色 / 中和是否到位 |
| 梯度残差(分块背景方差) | 自算 | DBE/ABE 去梯度效果 |

这些指标构成"骨架",大多数参数调整由它们驱动(见 5.2)。

### 4.2 感知评委(多模态 LLM,做审美方向)
处理数值指标覆盖不了的主观/感知问题。

- **输入**:标准化预览 PNG(整图 + 可选裁切块)+ 当前指标 JSON + 当前步骤上下文。
- **输出**:**结构化、离散、有界**的决策(严格 JSON),例如:
  ```json
  {
    "verdict": "needs_adjustment",
    "issues": ["green_cast", "over_sharpen_halo"],
    "actions": [
      {"target": "SCNR", "direction": "increase", "step": 1},
      {"target": "sharpen_strength", "direction": "decrease", "step": 1}
    ],
    "stop": false,
    "confidence": 0.7,
    "reason": "背景整体偏绿;亮星周围可见暗环。"
  }
  ```
- **动作集(离散)**:见 §5.2.1 定稿(方向×三档幅度 + stop/escalate/revert),禁止 LLM 直接输出连续参数值。
- **Prompt 设计原则**:
  - 明确告知"你在评审一张深空天体图的后期结果";评审维度直接采用 §6.4 的**过处理失败模式清单**(絮状纹理、伪细节/塑料感、暗部假光斑、缩星过度、背景发白、偏色、过饱和)。
  - 图**和**数一起给(数值让判断更稳)。
  - 强约束输出 schema;要求给 reason 便于日志与人工回看。
  - 每次只在**有界动作集**里选,低置信度时倾向 `keep`。
  - **真实性硬规则**(§6.5):评委只能建议"调参方向",禁止建议"补画/凭空添加信息"。
- **介入时机(省成本)**:不是每步都调 LLM;只在阶段切换、终止判断、以及定量层报告"疑似偏色/伪影"时介入(见 5.3)。

### 4.3 LLM 接入与模型配置(多厂商可选)
用户须能自选模型厂商,不锁定单一供应商。

- **供应商抽象层**:定义统一的 `VisionCritic` 接口(输入:图 + 指标 + prompt;输出:约定 JSON),
  下接各厂商适配器:Anthropic Claude、OpenAI、Moonshot Kimi、DeepSeek、以及 OpenAI 兼容端点(可接本地/自建)。
- **模型配置界面**:GUI 里提供设置页,用户填写「供应商 / 模型名 / API Key / Base URL / 温度」,可测试连通性,配置本地加密保存。
- **⚠️ 能力要求 — 视觉是硬门槛**:审美评委必须用**多模态(视觉)模型**。
  - 具备视觉:Claude(最新多模态)、GPT-4o 类、Kimi 视觉、DeepSeek-VL 等 → 可担任评委。
  - 纯文本模型:**不能**做图像评委;可降级为"仅基于定量指标"的顾问模式(能力受限,需在 UI 明确提示)。
- **推荐默认**:Claude 最新多模态模型;但默认可改,配置项持久化。
- **稳健性**:强制结构化输出(JSON schema / function calling);关键节点可选多次采样投票降低波动;
  调用失败/超时 → 回退到"纯定量决策"并记录,不中断管线。
- 🔶 待确认:首批适配哪几家(建议 Claude + OpenAI + 一个国内:Kimi 或 DeepSeek)?是否需要"同一决策多模型交叉验证"?

## 5. 编排/决策层

### 5.1 管线状态机
管线是一串"阶段"(见第 6 节),每阶段:执行 → 取指标+预览 → 评估 → 决策(调参重跑 / 通过 / 回退)。编排器维护当前状态、历史、可回退快照。

### 5.2 数值优化策略(每个可调参数)
对定量可判定的参数,用简单稳健的策略,不上重型优化:
- **单调收敛型**(如解卷积迭代、降噪强度):沿指标(FWHM/σ)方向逐档推进,指标不再改善或触发副作用阈值(星点畸变、剪切)即停。
- **目标值型**(如拉伸黑场 → 目标背景中位数、剪切 <0.1%):用少量迭代/二分逼近目标。
- **约束优先**:任何调整不得越过硬约束(如白场剪切上限);越界即回退。

### 5.2.1 动作粒度设计(定稿方案)
LLM 与数值层都不直接吐连续数值,而是在一套**离散、有界、粗到细**的动作空间里选择;
每个参数预先定义"档位表"把抽象动作映射成具体增量。

- **方向 × 幅度**(每参数):
  - 方向:`increase` / `decrease` / `keep`
  - 幅度:`slight`(≈±1 档,细调)/ `moderate`(≈±2 档,默认)/ `strong`(≈±3 档,粗调)
- **流程动作**:`stop`(本步收工)、`escalate`(降级到下一回退实现或转手动)、`revert`(回退上一步快照)。
- **粗到细收敛(自动挡内)**:首轮用 `moderate` 探方向 → 指标改善则同向继续,若**过冲**(指标反而变差)则**反向 + 幅度减半**(类似线搜索),逐步收敛到 `slight`;达到该步最大迭代数或无改善即 `stop`。
- **档位表由参数决定**:例如 BXT `sharpen` 的一档 = 0.05,NXT `denoise` 的一档 = 0.1,HT 黑场一档 = 目标背景中位数的一定比例——具体数值在 P2 标定,LLM/编排器只认"档",不认数。

> 设计理由:三档幅度足够覆盖"微调/常规/大改"三种意图,又不会像连续数值那样让 LLM 失稳;
> 粗到细 + 过冲减半保证收敛;所有"档→值"的映射集中在一张表里,便于调参与复现。

### 5.3 LLM 介入调度
```
每阶段结束:
  1. 定量层给指标 → 硬约束检查(越界→回退/调参)
  2. 若指标全绿且非关键节点 → 直接通过(不调 LLM)
  3. 若为阶段切换/终止判断/指标提示疑似主观问题 → 调 LLM 评委
  4. 合并"数值决策 + LLM 方向决策" → 生成下一步 job
```

### 5.4 终止条件
- 指标收敛(连续 N 步无实质改善)**且** LLM 判定 `stop:true`;或
- 达到该阶段最大迭代数(防死循环);或
- 触发不可恢复的硬约束(记录并停,交人工)。

### 5.5 人工介入点(HITL,已确认:v1 默认半自动)
- **默认半自动**:在主观/高风险节点(A4 拉伸、A3 颜色校准、阶段 B 全部配色/合成)暂停,展示预览 + 指标 + 系统建议,由用户确认 / 微调 / 让系统重试后再继续。
- 客观节点(A1 梯度校正、A5 降噪的数值收敛)默认自动通过,除非触发护栏。
- 提供"全自动"开关供熟练用户一键跑到底。
- **副产物**:半自动阶段用户的每次确认/否决/微调都记录下来,作为"用户偏好样本",为将来个性化 LLM 评委积累数据。

## 6. 处理管线定义(DSO 具体流程)

> 本节依据作者两篇后期文章的实际工作流重写。总纲遵循"四大问题"框架:
> ①消除瑕疵 ②提升信噪比 ③榨取细节 ④增强美感。①②由 WBPP 一键完成(能力已足够),
> 属**可选前置**,不是核心;系统核心是 ③④两个阶段。

### 6.0 项目初始化:拍摄配置声明(交互,必做)
配色与颜色校准的走向**不能由程序猜**,必须在项目开始时由用户声明。启动新项目时用一个向导询问:

1. **相机类型**:彩色(OSC) / 黑白(Mono)。
2. **拍摄/滤镜方案 → 处理路线**(据此决定 A3 颜色校准与阶段 B 配色分支):

   | 相机 | 常见方案 | 处理路线 | 说明 |
   |---|---|---|---|
   | OSC | 无窄带(RGB/宽带) | **真实色路线** | SPCC 还原真实色,美感阶段以真实为主 |
   | OSC | Hα/OIII 双窄带(如 L-eNhance/L-Ultimate) | **HOO 路线** | 最常见的 OSC 双窄带,**默认推荐** |
   | OSC | SII/OIII 双窄带 | **HOO 型双色**(SII→R,OIII→G/B) | 按双色思路映射,同 HOO 配方 |
   | OSC | SII/Hβ 双窄带 | **HOO 型双色**(SII→R,Hβ→G/B) | 因素较多,向导额外提示 |
   | Mono | LRGB | 真实色路线 | 分通道合成 |
   | Mono | 双窄带(Hα/OIII 等,2 通道) | **HOO 路线** | 只有 2 通道 → HOO,不能 SHO |
   | Mono | 三窄带(SII/Hα/OIII,3 通道) | SHO(可选)/ HOO | 有 3 通道才谈 SHO |

3. **目标配色**(仅窄带):
   - **双窄带 → 默认且推荐 HOO**。作者经验:双窄带只有 2 通道,做 SHO 会缺一个通道、不好看;HOO 呈红蓝配色更美观。
   - 仅当有**三窄带(SII/Hα/OIII)**时才提供 SHO 选项。
   - 允许自定义通道映射(高级)。

> ⚠️ 这一步是**必要的用户询问**,不可跳过默认。OSC 素材主流是 RGB 与 Hα/OIII 双窄带两条路线,
> 但也有 SII/OIII、SII/Hβ 等滤镜,考虑因素多,必须让用户提前决定处理方向。
> 双窄带的具体合成配方见 **§6.6**。

### 6.0.1 输入前提与前置(可选)
- 核心输入:**WBPP 叠加好的线性主图**(单通道 / RGB / 窄带合成前的分通道)。
- 可选前置:若用户提供亮/暗/平/偏置场,工具可调 WBPP 完成校准+叠加,但默认认为输入已是主图。
- **管线第一步必做:裁黑边**(DynamicCrop / 自动检测黑边)。作者明确:不裁黑边直接做梯度校正会"很离谱"。

### 6.1 阶段 A — 榨取细节(线性阶段)
每步 = "解决一个具体问题 + 一个可判定的达标标准",并按**自动→手动降级阶梯**执行。

| 步 | 目的 | 首选实现 | 降级/回退阶梯 🔻 | 达标标准(定量 / LLM) |
|---|---|---|---|---|
| A1 梯度校正 | 去光污染梯度,背景归中性灰 | GraXpert(AI 自动) | → GradientCorrection(GC) → DBE(手动) | 定量:分块背景方差↓、R/G/B 背景差小;LLM:背景平整无渐变、无残留光斑 |
| A2 反卷积 | 恢复清晰度、修星点 | **BXT**(缩星力度调小,更自然) | 无三件套 → 原生 Deconvolution(需 PSF)/ 或跳过 | 定量:FWHM↓、偏心率↓、星数基本不变;LLM:**无絮状/蚯蚓纹理**、星点无黑环 |
| A3 颜色校准 | 还原正确色彩 | ImageSolver 解析 → **SPCC**(Gaia DR3) | 无 SPCC → BackgroundNeutralization + ColorCalibration | 定量:通道背景差→0;LLM:背景中性、星色/星云色自然;**放在 A2 之后做** |
| A4 拉伸 | 线性→非线性 | STF(仅预览)+ **HT** | GHS(高阶,可选) | 定量:黑场剪切<阈值、白场不烧核、背景中位数达目标;LLM:核心不过曝、背景不发白 |
| A5 降噪 | 抑制拉伸后噪点 | **NXT**(黑白片不勾 chroma) | 无 NXT → MultiscaleLinearTransform / TGVDenoise | 定量:背景σ↓;LLM:**无塑料感 / 无伪细节**、细节保留 |

> 降级触发:某实现执行后评估不达标(定量越界或 LLM 判"不理想"),自动降到阶梯下一级重试。这是把作者
> "GraXpert 不行就 GC,再不行就 DBE"的习惯抽象成的**通用自适应模式**(§6.3)。

### 6.2 阶段 B — 增强美感(非线性阶段,LLM 主导)
进入主观创作,LLM 评委权重更高。

1. **星点分离**:SXT(首选)→ 无则 StarNet(免费)/ StarMask+形态学缩星。产出 `starless` + `stars`。
2. **星云/星系主体**(starless):CurvesTransformation 调饱和/对比/色彩;SCNR 去绿。
3. **星点**(stars):CurvesTransformation 提星点饱和度。
4. **合成**:PixelMath 把 stars 合回 starless。
5. 成片导出 + 全程决策日志。

**配色分支** 🔶:
- **宽带**:以还原真实为主,偏离真实色高风险(作者经验:翻车概率大)。
- **窄带**:可创意配色(哈勃 SHO 等伪色允许),约束放宽。

### 6.3 通用自适应模式(贯穿全线)
两个维度的自适应,复用同一套编排逻辑:
- **能力自适应**:启动时探测 BXT/SXT/NXT/StarNet/GraXpert/Gaia DR3 是否可用,为每步选择可用的最高档实现。
- **质量自适应(降级阶梯)**:同一步内,自动实现不达标 → 升级到更强/更手动的实现重试;仍不达标 → 记录并交人工(§5.5)。

### 6.4 过处理护栏(评委检查清单,源自作者列举的失败模式)
既作为 LLM 感知检查项,能量化的同时落为硬约束:
| 失败模式 | 成因 | 硬约束(定量) | LLM 感知检查 |
|---|---|---|---|
| 絮状 / 蚯蚓纹理 | 反卷积/锐化过度 | FWHM 异常、局部高频突增 | ✔ 纹理是否虚假 |
| 伪细节 / 塑料感 | 降噪过度 | 细节能量骤降 | ✔ 是否过度平滑 |
| 暗部光斑误当星云 | 拉伸过度 + 梯度残差 | 背景方差、剪切 | ✔ 暗部是否有假结构 |
| 星点过度缩小/不自然 | BXT 缩星过强 | 星数下降、星径异常 | ✔ 星点是否自然 |
| 背景发白/发灰 | 拉伸过度 | 背景中位数越界 | ✔ 背景观感 |
| 偏色 | 校准不足 | 通道背景差 | ✔ 整体色偏 |
| 过饱和 | 曲线过度 | 饱和统计 | ✔ 是否俗艳失真 |

### 6.5 真实性原则(硬规则)
作者明确反对"涂抹上色/凭空加信息"。系统**只做基于已有信号的拉伸/增强,不生成不存在的结构**;
LLM 评委只输出参数方向,不得建议"补画/添加"。此为不可协商的硬规则。

### 6.6 双窄带 HOO 合成配方(作者流程,定稿)
双窄带默认走 HOO。合成发生在阶段 A 拉伸(A4)前后,替换掉常规 A3 颜色校准,插入到主管线中。

**通道准备(因相机而异)**
- **Mono 双窄带**:已有独立 Hα、OIII 两个通道主图,直接进入下一步。
- **OSC 双窄带**:先拆分 RGB 三通道 → `Hα = R`,`OIII = (G + B)`(G、B 相加)。用 PixelMath/ChannelExtraction 实现。

**逐通道拉伸(线性→非线性,用 HT)**
1. 解除默认拉伸(复位 STF,回到线性真实状态再手动处理)。
2. 对每个通道用 **HistogramTransformation** 手动拉伸,达标标准(作者定量):
   - **直方图亮度峰值** 落在拉伸面板约 **1/8** 处;
   - **背景亮度**(用 CurvesTransformation 探针在背景取值)≈ **16145 / 65535 ≈ 0.246**(16-bit 值)。
   - 这两个是本步的定量目标,由 §5.2.1 的档位逼近;半自动下暂停给用户核对。

**合成 HOO**
3. 通道映射:`R ← Hα`,`G ← OIII`,`B ← OIII`,其中 **OIII = G + B(原始 G、B 通道相加)**。
   即输出 G、B 两通道都等于 OIII 信号(=原 G+B)。用 PixelMath 合成 RGB。
4. **去绿**:若合成后偏绿,用 SCNR(或等效)做适度去绿。
5. 目标观感:整体**红蓝配色**,协调美观(作者审美基准)。

> SII/OIII、SII/Hβ 走同一配方,仅通道来源不同(SII→R;OIII 或 Hβ→G/B)。
> 达标数值(1/8、16145)先作为默认基准,后续按目标类型可微调。

### 6.7 窄带 + RGB 混合(星点还原,可选增强)
当用户**同时拍了窄带与 RGB**时,提供作者偏好的混合路线,让星点色彩更自然:
1. 从 **RGB 主图** 用 SXT/StarNet 分离出**星点图(RGB stars)**。
2. 窄带图(HOO 等)按主管线处理并**去星**,得到 starless 窄带星云。
3. 用 PixelMath 将 **RGB 星点** 叠加回 **去星窄带星云**(屏幕/相加混合)。
- 效果:窄带星云的层次 + RGB 星点的真实色,观感更美。
- 依赖:需要星点分离能力(SXT 或 StarNet);无则提示该增强不可用。

> 🔶 待确认:§6.1 主干顺序与首选工具是否还有要调?§6.6 的 1/8、16145 是否要按不同目标(如亮星云 vs 暗星云)给不同默认?

### 6.7.0 星点拉伸策略(作者两法,已采用策略2)
对"分离出的星点图"直接做背景归一化拉伸会炸开噪声(星点图背景近 0,拉伸把黑背景抬起 → 放大固定
模式噪声/棋盘纹、星点膨胀)。作者的两种正确做法:
1. **先合拉再分离**:星点与星云一起拉伸 → 再分离星点 → 单独调星点饱和/色彩。结果自然。
2. **STF 套用(已采用)**:用**全图**算 STF,分离星点后把该 STF 套到星点图(线性→非线性)。
   星点图背景近 0 会落在曲线黑场之下自动压黑,星点以"真实背景为基准"的合理曲线提亮;
   星云则保持线性,可单独更狠地拉伸。**暗目标首选此法**。
   → 实现:`stretch` op 的 `stfFrom=<全图路径>` 参数。

### 6.7.1 窄带星点处理配方(作者流程,暗目标默认用窄带星点)
本项目(船底座暗弱目标)先用**窄带星点**(不走 §6.7 的 RGB 星点混合)。
SXT 分离出星点图后,对 stars 单独处理:

1. **拉伸星点**:保证星点亮度(与 starless 分开拉,力度更克制,§6.8)。
2. **提升星点饱和度**:CurvesTransformation 提饱和,让星色更鲜明。
3. **星点去绿**:SCNR 去绿;
   - **发紫星点特例**:先 `Ctrl+I` 反相 → SCNR 去绿 → 再 `Ctrl+I` 反相(反相后洋红变绿,去掉再反回)。
     对应 PixInsight:Invert → SCNR → Invert。
4. **去绿强度技巧(重要默认)**:SCNR **不要全量去绿**(amount=1.0 观感不自然),
   作者常用 **amount ≈ 0.7~0.8**,更自然。系统默认 **0.75**,并在半自动节点由用户按预览微调。

> 这条配方同样体现在 starless 星云的去绿上:SCNR 默认 0.75 而非 1.0。

### 6.8 目标相关策略(经验规则,影响管线分支)
不同目标的特性会改变管线的侧重与参数,需要在项目配置中携带"目标画像"并据此调整:

- **暗弱目标(如超新星遗迹、暗星云)→ 星点/星云分开拉伸(强约束)**:
  暗目标若与星点一起拉伸,拉亮星云就会烧星点、顾星点则星云出不来。因此对暗目标,
  **必须先用 SXT 分离星点**,对 starless 做**更激进的拉伸**(更低 targetBackground / 更大力度),
  对 stars 做**克制的拉伸**,最后 PixelMath 合成。这是暗目标的**默认策略**,非可选。
  → 拉伸顺序调整为:(反卷积后)先 starsep → 分别 stretch → recombine。
- **低空/高光害拍摄 → 梯度更重**:如北半球拍南天低空目标(船底座等),叠加后梯度明显,
  梯度校正可能需要**多次**或优先用 GraXpert(AI 重建背景),GC/DBE 兜底。
- **亮目标(如 M42 核心)→ 注意高光保护**:拉伸/合成时优先防止核心过曝,可用 HDR/分区处理。

> 落地:项目配置(§6.0)增加"目标亮度类型"(暗弱/中等/明亮)与"拍摄条件"(是否低空/重光害)标签,
> 编排器据此选择拉伸策略与梯度校正强度。

## 11. 风险与对策
| 风险 | 对策 |
|---|---|
| PixInsight 非真 headless / 会话稳定性 | 常驻 job-runner + 心跳;崩溃自动重启会话并从快照续跑 |
| LLM 幻觉/波动 | 离散有界动作集、图+数并给、低温度、可选多次投票、低置信度默认 keep |
| 预览渲染不一致导致评估漂移 | 固定 STF/导出参数,记录并复现 |
| 数值优化死循环 | 每阶段最大迭代数 + 无改善早停 |
| 过处理(塑料感/伪影) | 硬约束(星点畸变、剪切、星数下降)+ LLM 伪影检查 |
| 成本/延迟 | LLM 仅关键节点介入;定量层承担多数决策 |
| 不可复现 | 全程 job/result/决策落盘,固定随机性/参数版本 |
